# Нефункциональные требования (NFR)

> Проект: LLM Agent Platform
> Дата: 2026-03-28
> Домен: AI Infrastructure / API Gateway для LLM

---

## 1. Производительность (Performance Efficiency)

### 1.1 Профиль нагрузки
- **Целевые пользователи:** Преподаватель/проверяющий + нагрузочный тест
- **Одновременные сессии (нормальная работа):** 10-15
- **Одновременные сессии (пик):** 20-30
- **Стресс-тест:** 40-50 (провокация отказов для демонстрации устойчивости)

### 1.2 Пропускная способность (Throughput)
- **Платные модели:** до 19 RPS (лимит OpenRouter = баланс $19)
- **Бесплатные модели:** 20 RPM (Step 3.5 Flash, Nemotron 3 Super)
- **Целевой throughput нагрузочного теста:** 50-100 запросов серией

### 1.3 Время отклика (Latency)
- **Overhead платформы (без учёта LLM):** < 50ms (p95)
- **TTFT (Time to First Token):** метрика, не целевое значение - зависит от провайдера
- **TPOT (Time per Output Token):** метрика, не целевое значение - зависит от провайдера
- **Health-check endpoint:** < 10ms

### 1.4 Streaming
- **Обязательно:** SSE (Server-Sent Events) для поточной передачи ответов LLM
- **Требование:** Балансировщик не буферизует поток, передаёт токены по мере получения

## 2. Надёжность (Reliability)

### 2.1 Доступность (Availability)
- **SLA:** Не применимо (учебный проект, локальный деплой)
- **Требование:** Платформа остаётся работоспособной при отказе отдельных LLM-провайдеров

### 2.2 Отказоустойчивость (Fault Tolerance)
- **Circuit Breaker:** При получении 5xx/429/timeout от провайдера - временное исключение из пула
- **Fallback:** Автоматический переход на другого провайдера при отказе текущего
- **Cooldown:** Исключённый провайдер возвращается в пул через настраиваемый интервал
- **Демонстрация:** Нагрузочный тест должен показать корректную деградацию при достижении rate limit

### 2.3 Восстановление
- **RPO:** Не применимо (in-memory, stateless)
- **RTO:** Перезапуск Docker Compose (~30 секунд)

## 3. Безопасность (Security)

### 3.1 Регуляторные требования
- Нет (учебный проект)

### 3.2 Аутентификация и авторизация (Уровень 3)
- **API-токены:** Bearer tokens для авторизации запросов к платформе
- **Агенты:** Каждый зарегистрированный агент получает токен при регистрации
- **LLM-провайдеры:** API-ключ OpenRouter хранится на стороне платформы, не передаётся клиентам

### 3.3 Guardrails (Уровень 3)
- **Prompt injection detection:** Фильтр входящих запросов на типичные паттерны инъекций
- **Secret leakage prevention:** Фильтр исходящих ответов на утечку API-ключей, токенов, паролей
- **Конфигурируемость:** Guardrails включаются/выключаются через конфиг

### 3.4 Секреты
- **OpenRouter API key:** Переменная окружения, не в коде
- **.env файл:** В .gitignore, .env.example с плейсхолдерами в репозитории

## 6. Наблюдаемость (Observability)

### 6.1 Метрики (Prometheus)
- Количество запросов (total, по провайдерам, по моделям)
- Латентность: p50, p95 (общая и overhead платформы)
- TTFT (Time to First Token) по провайдерам
- TPOT (Time per Output Token) по провайдерам
- Количество входных/выходных токенов
- Стоимость запроса (расчётная, на основе цен провайдера)
- Коды ответов (2xx, 4xx, 5xx)
- Загруженность CPU
- Состояние провайдеров (healthy/unhealthy)

### 6.2 Дашборды (Grafana)
- **Обязательные:**
  - Латентность по провайдерам (p50/p95, временной ряд)
  - Распределение трафика по провайдерам (pie chart / stacked bar)
  - Коды ответов (временной ряд)
  - TTFT/TPOT по моделям
  - Стоимость запросов (кумулятивная)
- **Health:** Статус провайдеров, circuit breaker состояния

### 6.3 Трассировка (OpenTelemetry + Langfuse)
- **OpenTelemetry:** Distributed tracing для каждого запроса через платформу
- **Langfuse:** Трассировка работы агентов - входы, выходы, промежуточные шаги, tool calls

### 6.4 Логирование
- **Формат:** Structured JSON logs
- **Уровень:** INFO по умолчанию, DEBUG через переменную окружения
- **Хранение:** stdout (Docker logs), без внешнего хранилища

### 6.5 Health Checks
- **Каждый сервис:** GET /health возвращает статус + зависимости
- **Провайдеры:** Периодический health check (ping с минимальным запросом или проверка доступности endpoint)

## 7. Совместимость (Compatibility)

### 7.1 Клиенты
- **API:** REST (JSON), OpenAI-совместимый формат запросов/ответов
- **Streaming:** SSE (text/event-stream)
- **Браузер/UI:** Нет (API only), Grafana dashboards через браузер

### 7.2 Сеть
- **Доступ:** Локальный (localhost), Docker network
- **Внешний доступ:** Только к OpenRouter API (исходящий HTTPS)

### 7.3 Интеграции
- **OpenRouter API:** Единственный внешний LLM-провайдер
- **A2A Protocol:** Межагентная коммуникация внутри платформы

## 8. Жизненный цикл данных (Data Lifecycle)

### 8.1 Хранение
- **Реестр агентов и провайдеров:** In-memory (теряется при перезапуске)
- **Метрики:** Prometheus (retention по умолчанию - 15 дней)
- **Трассировки:** Langfuse (локальное хранилище в Docker volume)

### 8.2 Персистентность
- **Не требуется** для основных данных
- **Docker volumes** для Prometheus, Grafana, Langfuse (сохраняются между перезапусками контейнеров)

## Сводка аналитика

### Главные вызовы
- **Rate limits OpenRouter** - основное ограничение для нагрузочного тестирования. Бесплатные модели: 20 RPM. Платные: 19 RPS при текущем балансе. Нагрузочный тест должен быть спроектирован с учётом этих лимитов.
- **Streaming через балансировщик** - прокси должен передавать SSE-поток без буферизации, это нетривиальная задача для async-фреймворка.
- **Объём за 1 день** - три уровня сложности с наблюдаемостью, guardrails, нагрузочными тестами. Критически важна параллелизация работы.

### Серые зоны
- Точные пороги circuit breaker (количество ошибок до исключения, cooldown) - определятся при реализации
- Формат Agent Card для A2A - уточнить по спецификации Google A2A protocol
- Конкретный набор автономных агентов для демонстрации - определится при планировании

### Рекомендации
- Использовать OpenAI-совместимый формат API (OpenRouter его поддерживает) - упрощает интеграцию
- Для нагрузочного теста: сначала бесплатные модели (показать throughput), затем дешёвые платные (показать масштаб), в конце стресс-тест до 429
- Guardrails реализовать как middleware (pipeline), чтобы легко включать/выключать
- Агенты с tool use: реализовать 2-3 агента с разными инструментами (web search, calculator, code execution) для наглядности
